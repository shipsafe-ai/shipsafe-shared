# ShipSafe · Universal CLI Setup Architecture

## The promise

Every ShipSafe submission installs, configures, and runs
with a single command. No manual setup. No reading docs first.
No "step 3: configure 14 environment variables."

```bash
npx shipsafe-[agent] init
```

That is the entire getting started guide.

## Universal CLI package structure

Each agent is an independent npm package published to npm:

```
@shipsafe/cargodb      → MongoDB track
@shipsafe/routeforge   → GitLab track
@shipsafe/voyageblack  → Elastic track
@shipsafe/tidesync     → Fivetran track
@shipsafe/naviguard    → Arize track
@shipsafe/agentops     → Dynatrace track
```

All installed via npx — no global install required.

## CLI commands — same across all 6

```bash
# Initialize and deploy
npx shipsafe-[agent] init

# Run the built-in demo scenario (ShipSafe crisis)
npx shipsafe-[agent] demo

# Connect to your own data source
npx shipsafe-[agent] connect

# Check status of deployed agent
npx shipsafe-[agent] status

# View live logs
npx shipsafe-[agent] logs

# Destroy deployment
npx shipsafe-[agent] destroy

# Update to latest version
npx shipsafe-[agent] update

# Run health check
npx shipsafe-[agent] health
```

## What each command does internally

### `init`

```typescript
// packages/cli/src/commands/init.ts
export async function init(agentName: string) {
  await checkSystemRequirements()     // node, gcloud CLI, docker
  await detectExistingServices()      // already have GCP project?
  await promptConfiguration()         // minimal questions only
  await validateConnections()         // test all credentials
  await deployToCloudRun()           // terraform apply
  await configureGemini()            // vertex AI setup
  await configureMCP()               // partner MCP server
  await runHealthCheck()             // verify everything works
  await printSuccessSummary()        // show URLs and next steps
}
```

### `demo`

```typescript
// Loads pre-built ShipSafe scenario data
// No external dependencies — works offline
// Seeds the connected data source with demo data
// Opens the dashboard
export async function demo(agentName: string) {
  await seedDemoData(agentName)       // loads ShipSafe fixtures
  await triggerCrisisScenario()      // starts the Hormuz incident
  await openDashboard()              // opens browser
}
```

### `connect`

```typescript
// Connects agent to user's own data source
// Replaces demo data with real data
export async function connect(agentName: string, options: ConnectOptions) {
  await validateDataSource(options.uri)
  await updateConfiguration(options)
  await testConnection()
  await printConnectionSummary()
}
```

## The configuration system — zero hardcoding

Every agent uses a single config file generated by `init`:

```yaml
# ~/.shipsafe/cargodb.yaml (generated, never edit manually)

agent: cargodb
version: 1.0.0
deployed_at: 2026-05-14T14:55:00Z

gcp:
  project_id: my-project-id
  region: us-central1
  cloud_run_url: https://cargodb-abc123-uc.a.run.app

gemini:
  model: gemini-1.5-pro
  location: us-central1

partner:
  type: mongodb
  uri: mongodb+srv://...     # stored in GCP Secret Manager
  database: production        # user-specified during init

demo:
  mode: false                 # true = use ShipSafe data
  scenario: hormuz_crisis     # which demo scenario to run
```

The agent reads this config at startup. Switching from demo to
production is one flag change — or one `connect` command.

## Per-agent init details

### CargoDB (MongoDB)

```bash
npx @shipsafe/cargodb init

Questions asked:
  1. MongoDB connection string (or Atlas API key)
  2. GCP project ID
  3. Which collections to monitor (or "all")
  4. Alert webhook URL (Slack/email) — optional

What gets deployed:
  - Cloud Run service: cargodb API + dashboard
  - GCP Secret Manager: MongoDB credentials
  - Vertex AI: Gemini 1.5 Pro configured
  - MongoDB MCP: connected to user's cluster
  - Scheduler: daily schema health report at 08:00

Works with any MongoDB database. Not just shipping data.
Demo uses ShipSafe port data. Production uses whatever you connect.
```

### RouteForge (GitLab)

```bash
npx @shipsafe/routeforge init

Questions asked:
  1. GitLab personal access token
  2. GitLab project ID (or URL)
  3. GCP project ID
  4. Risk threshold for auto-blocking (default: 75/100)
  5. Post comments on MRs automatically? (yes/no)

What gets deployed:
  - Cloud Run service: routeforge API + dashboard
  - GitLab webhook: configured automatically on the project
  - GCP Secret Manager: GitLab token
  - Vertex AI: Gemini 1.5 Pro configured
  - GitLab MCP: connected to user's project

Works with any GitLab project. Not just routing algorithms.
Any code change to any critical business logic benefits from this.
Demo uses ShipSafe routing algorithm MRs.
```

## The demo data system

Every agent ships with built-in demo data. No external dependencies.

```
packages/demo-data/
├── src/
│   ├── scenarios/
│   │   ├── hormuz-crisis.ts    # primary demo scenario
│   │   ├── schema-conflict.ts  # CargoDB demo
│   │   ├── bad-algorithm-mr.ts # RouteForge demo
│   │   ├── incident-window.ts  # VoyageBlack demo
│   │   ├── sync-failure.ts     # TideSync demo
│   │   └── model-regression.ts # NaviGuard demo
│   ├── fixtures/
│   │   ├── vessels.json        # 15 real vessel records
│   │   ├── ports.json          # 12 real port records
│   │   ├── cargo.json          # 200 container manifests
│   │   ├── logs.json           # 30 days of Elastic logs
│   │   └── traces.json         # 500 Arize model traces
│   └── seed.ts                 # seeds any connected service
└── package.json
```

Running `npx shipsafe-[agent] demo` seeds the connected service
with this data and triggers the crisis scenario.

Running `npx shipsafe-[agent] connect` replaces demo data with
the user's real data and disables demo mode permanently.

## Universality checklist — every submission must pass

Before any submission is finalized, verify:

- [ ] `npx shipsafe-[agent] init` runs end-to-end in < 3 minutes
- [ ] `npx shipsafe-[agent] demo` works with zero configuration
- [ ] `npx shipsafe-[agent] connect` works with a real external account
- [ ] README makes zero mention of shipping/logistics in the first 3 sections
- [ ] README explicitly states the agent works for any domain
- [ ] The agent runs against a non-shipping test case successfully
- [ ] All credentials stored in GCP Secret Manager (nothing in .env files)
- [ ] MIT license visible in GitHub repo About section
- [ ] One-click deploy button works from a fresh GCP project
